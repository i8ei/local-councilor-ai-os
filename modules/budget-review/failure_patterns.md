# 予算PDF・SQLite化の失敗パターン

| 失敗 | 何が起きるか | 防ぎ方 |
|---|---|---|
| 当初予算と補正後予算を混ぜる | 増減の意味が崩れる | `budget_stage` を必須にする |
| 補正額を累計額として読む | 補正前後の検算が壊れる | `pre_supplement_amount + supplement_amount = post_supplement_amount` を検算する |
| 歳出だけ見て歳入・財源を見ない | 財源確認の問いが抜ける | 歳入歳出のtotal行を両方入れ、同額を確認する |
| 主要事業説明書だけで判断する | 款項目節の予算構造とずれる | 事項別明細書の行へ戻れる `source_locator` を残す |
| 前年度比較列を自前計算だけで埋める | 原表の比較列と照合できない | `raw_value` に原表行を残し、比較列を検算する |
| 一般会計と特別会計を混ぜる | 予算規模が崩れる | `account_name` ごとにDB行と検算を分ける |
| 千円と円を混同する | 1000倍ずれる | `unit` を原表どおり保存し、暗黙換算しない |
| 新規事業と名称変更を混同する | 実態を誤る | 名称一致だけで年度間対応を確定しない |
| 款項目節の階層を取り違える | 合計が合わない | `grain` とコード必須条件で停止する |

不一致が出た場合は、合計に合わせて数字を直さず、原典、入力行、単位、階層、会計、予算段階を確認する。
