# 決算PDF・CSV化の失敗パターン

| 失敗 | 何が起きるか | 防ぎ方 |
|---|---|---|
| `千円`を`円`として読む | 1000倍ずれる | `unit` を原表どおり保存し、暗黙換算しない |
| PDFページと印刷ページを混同する | 原典位置へ戻れない | `pdf_page` と `print_page` を分ける |
| 目合計を節ごとに重複加算する | 歳出款合計が膨らむ | 節行の目合計は反復値として保存し、集計時に目粒度で重複除去する |
| ハイフンをゼロ扱いする | 欠測、非該当、秘匿を誤る | `raw_value` を残し、意味未確認なら公開利用しない |
| OCRが数字を誤読する | 合計が合わない、または偶然合っても内訳が壊れる | 差額ゼロ検算と原典スポット確認を併用する |
| 補正後予算と当初予算を混ぜる | 比較定義が崩れる | `definition` に列意味を明記する |
| 一般会計と特別会計を混ぜる | 会計範囲が崩れる | `account_name` を必須にし、会計ごとに検算する |
| 歳入と歳出の列名を流用する | 金額列の意味が変わる | `summary`、`revenue`、`expenditure` を分ける |
| 合計が合うように数字を直す | 原典とDBの対応が壊れる | 不一致時は原典、CSV、抽出ログを確認し、推測補正しない |
| 非公開資料を公開資料と同じDBに混ぜる | 外部成果物へ内部情報が漏れる | `public_private_boundary.md` に従い、保存先と利用可否を分ける |

## 不一致が出た時の扱い

不一致は失敗ではなく、検算ゲートが機能している状態である。次の順に確認する。

1. 単位が表ごとに違わないか。
2. PDFページ、印刷ページ、表名が合っているか。
3. 款、項、目、節の粒度を取り違えていないか。
4. OCRまたはAIが桁、符号、空欄を誤読していないか。
5. 反復された目合計を二重に足していないか。
6. 会計や年度が混ざっていないか。

原因が解消できない行は `reconciled` にしない。外部利用する場合は、その行を外すか、未確認として断定を避ける。
