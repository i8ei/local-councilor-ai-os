# 決算PDF抽出ガイダンス

## 基本方針

本リポジトリは、任意の決算PDFから正確に数字を読む汎用抽出器を提供しない。PDFの版面、会計区分、公開範囲、表頭、単位、ページ構成は自治体ごとに異なる。抽出は利用者のローカル環境で、AI、人手、OCR、`pdftotext`、表抽出ツールなどを組み合わせて行う。

共通化するのは、抽出後に `sqlite_input_contract.md` の形へ落とし、SQLiteへ投入し、`verify_totals.py` を通すことである。CSVはSQLiteへ渡すための中間入力形式であり、作業の中心は投入後のSQLiteである。

## 抽出AIへの依頼方針

抽出AIには、分析や判断ではなく、転記と構造化だけを依頼する。

```text
次の決算書PDFから、指定ページ範囲の表を読み取り、summary/revenue/expenditure のSQLite入力契約に合わせて、取込用CSVの行を作成してください。

禁止:
- 読めない数字を推測しない
- 空欄、ハイフン、ゼロを同一視しない
- 単位を勝手に換算しない
- 小計や合計を自作しない
- 原表にない原因説明や評価を加えない

必須:
- raw_value に原表の該当行またはセル群を残す
- unit は原表の単位をそのまま記録する
- print_page と pdf_page を分ける
- source_locator に表名、ページ、行、列をJSONで残す
- 読めないセルは空欄にし、要確認として報告する
```

## 人が先に決めること

抽出前に、人が次を指定する。

1. 対象資料が公式公開資料か、内部資料か。
2. 対象年度、会計名、会計範囲。
3. 読む表のページ範囲。
4. 原表単位。
5. summary / revenue / expenditure のどれへ入れるか。
6. PDFページと印刷ページの対応。

AIへ丸ごとPDFを渡して「決算DBを作って」と依頼しない。ページ範囲と表種別を小さく区切る。

## 抽出後の検査

1. SQLite入力契約の必須列が埋まっているか確認する。
2. `ingest_csv.py` でSQLiteへ投入する。
3. `verify_totals.py` を実行する。
4. 不一致があれば、合計に合わせて数字を直さず、原典ページへ戻る。
5. 合格後、証拠台帳または判断ノートへ、検算日、対象資料、未解決事項を記録する。

## 合格しても分からないこと

差額ゼロ検算は転記と階層整合の確認であり、原因や妥当性を証明しない。不用額が大きい、繰越がある、収入未済がある、という事実は言えるが、その理由や政策評価は担当課説明、監査意見、事業資料など別根拠が必要である。
