# 01 案件開始

## 要旨

案件開始では、入口となった気づきを一つの問いと成果物へ絞り、自治体が答えられる権限範囲を確定し、調査の背骨となる案件ノートを作る。AIに調査を始めさせる前に、何を作る案件か、何を扱わないか、公開と内部のどちらで扱うかを人が決める。案件ノートは現在判断の正本であり、チャット履歴を正本にしない。

## 発動条件

住民の声が公開情報で検証できる問いに変換されたとき、予算または決算に異常の兆候を見つけたとき、計画案を検討するとき、自ら現場で課題を観察したとき、報道または制度変更から自治体への影響を確認する必要が生じたときに発動する。単なる関心ではなく、確認したい問いか監視したい変化が一文で言えることを入口とする。

## 入力と前提

- 起点となる問い、資料、観察記録のいずれか
- 想定する成果物と期限。成果物は一般質問、委員会質問、資料要求、公開説明、監視のみから選ぶ
- 内部原典を含む場合の取扱条件
- 案件を置くMOCと採番可能な`case_id`
- [正本主義](../way-of-working/01-canonical-notes.md)と[MOCとライフサイクル](../way-of-working/02-moc-and-lifecycle.md)に沿って保存する前提

## 手順

1. **【人】案件を「何を明らかにし、何を作るか」の一文で定義する。**成果物を一般質問、委員会質問、資料要求、公開説明、監視のみのいずれかに固定し、複数ある場合は主成果物を一つ決める。例：「事業実績の変化を確認し、委員会質問の前提を作る」。
2. **【人】権限を確認する。**論点が自治体の固有事務か、都道府県または国の権限か、共同処理かを法令、例規、事務分掌などで確かめる。自治体が持たない権限へ向けた案件は、行政が答えられない質問を生む。結論、法的根拠、確認日、原典位置を最初の証拠項目として記録し、不明なら`要確認`のまま次の行動を権限確認に限定する。
3. **【AI】案件の骨格案を作る。**既知の事実、仮説、未知の事項、候補原典を分け、各項目に根拠の有無を示す。AIが補完した内容は事実にせず、根拠不明は`不明`とする。
4. **【人】初日から情報区分を決める。**公開情報だけで構成するか、内部原典を含むかを記録し、迷う場合は内部とする。内部原典を含む案件から公開成果物へ逆引き可能な識別子を出さない。
5. **【併走】[案件サマリー](../templates/case-summary.md)から案件ノートを作る。**目的、成果物、権限の結論、初期仮説、次の行動を記入し、`status`を`active`にする。AIが記入漏れとリンク候補を点検し、人が内容を確定する。同じ操作でMOCへ役割付きの一行を追加し、本文から関連ノート一〜三本へwikilinkを張る。保存とリンクを別作業にしない。
6. **【人】対象外を明記する。**今回は扱わない地域、期間、制度、個別事情、成果物を列挙し、対象外へ広がる兆候が出たら別案件にする。対象外は放棄ではなく、現在の案件を完了可能にする境界である。

## プロンプト断片

```text
次の案件について、既知の事実、仮説、未知の事項、候補となる公式原典を分けて案件骨格を作成してください。根拠が与えられていない内容は推測で補わず「不明」と記してください。政策判断や優先順位は決めないでください。

案件の一文:
想定成果物:
確認済みの権限範囲:
入力資料:
```

```text
次の案件ノート案を点検してください。成果物、権限の結論と法的根拠、初期仮説、対象外、次の行動が明示されているかを確認し、不足項目だけを返してください。文章の完成度ではなく、着手可能性を評価してください。
```

## 成果物と完了条件

[案件サマリー](../templates/case-summary.md)に基づく`active`な案件ノートが存在し、主成果物、権限の結論と法的根拠、初期仮説、対象外、次の行動が記録されていること。MOCの一行と関連wikilink一〜三本があり、次の担当者がチャットを読まずに着手できれば完了とする。

## 次のワークフローへの接続

住民の声または現場観察をまだ問い化していない場合は[02 現場受付](02-field-intake.md)へ戻す。公開情報で検証できる問いが確定している場合は[03 公開調査](03-public-research.md)へ進む。調査不要の監視案件は、案件ノートに再開条件と次回確認日を置いて待機する。

## アンチパターン

- **成果物を決めずに検索を始める。** 資料の量だけが増え、どの確認で調査を終えるか決められない。
- **自治体の権限を推測で置く。** 行政が答えられない問いを作り、制度説明だけで質問が終わる。
- **AIの骨格案をそのまま正本にする。** 推測と本人の判断が混ざり、採用理由を説明できない。
- **内部と公開の区分を後回しにする。** 保存先、AI入力、公開リンクの境界が途中で崩れる。
- **保存だけしてMOCへつながない。** 次回の検索で案件の入口を失い、同じ調査を作り直す。
