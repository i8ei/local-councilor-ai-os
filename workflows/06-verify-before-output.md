# 06 公開前検算

## 要旨

質問通告または外部公開の直前に、結論を支える荷重主張を原典まで戻して検算し、出力してよいかを議員本人が決める。運用は「通告前」と「公開前」の二つのプロファイルに分ける。確認できない主張は断定を弱めるか外す。これは失敗ではなく、根拠に表現を合わせる正当な完了である。考え方は[検算ゲート](../way-of-working/07-verification-gates.md)に従う。

## 発動条件

- 質問通告を提出する直前
- 配布資料、ウェブ、SNS、メールなどを外部へ出す直前
- 根拠、数値、引用または公開範囲を変更した後の再承認時

## 入力と前提

- 検算対象となる版を固定した成果物
- 成果物に対応する証拠台帳と原典位置
- [Authority Router](../way-of-working/04-authority-router.md)が示す用途別出典と鮮度条件
- 「公開前」では想定読者、公開先、添付、リンク先が確定していること

証拠台帳にない主張を、原稿中の出典らしい表記だけで検証済みと扱わない。AIは抽出と照合を担うが、荷重主張の選定と最終承認は行わない。

## 手順

1. **【人】プロファイルと対象を宣言する。** 「通告前」または「公開前」を選び、成果物名、固定版、用途、提出先または公開先、予定日を記録する。両方に該当する場合は通告前を先に実施し、公開時に公開前を別記録で実施する。

2. **【AI】主張を機械抽出する。** 全文から数値、日付、固有名詞、制度上の主張、引用を漏れなく抜き、現在付いている出典注記と対応させる。主張、種別、出典注記の有無、`as_of`、`evidence_id`を表にする。`evidence_id`がない項目、ルーターの鮮度条件を超えた項目、本文と根拠で定義が一致しない項目を警告する。この段階で重要度を判定させない。

3. **【人】荷重主張を指定する。** 抽出表と見出し、結論を照合し、「誤りなら結論、質問の前提または主要な見出しが崩れる主張」に印を付ける。AIが候補を列挙してもよいが、採否は議員本人が決める。数値がない制度要件や引用も対象に含める。

4. **【併走】荷重主張を原典へ戻して裁定する。** 各主張について、公式索引から原典を開き、記載位置、版、`as_of`、定義、単位、引用の前後、必要な計算を人が画面で確認する。AIには該当箇所の対照と差分表示をさせる。結果を「検証済み」「断定回避に弱める」「外す」のいずれかにする。例：因果を確認できない場合は「影響した」から「同時期に変化した」へ弱める。

5. **【AI】公開前の漏えい走査を行う。** 「公開前」プロファイルでは、個人情報、内部ID、内部リンク、ファイル名、コメントや変更履歴などのメタデータ、文脈の組合せによる再識別可能性を検出する。問題の種類、位置、該当断片、要確認理由を一覧で返させる。削除可否は人が原文と公開目的に照らして決める。

6. **【人】最終の可否を記録する。** [公開前レビューテンプレート](../templates/public-output-review.md)から記録を作り、日付、プロファイル、対象版、荷重主張表、未解決事項、`go / no-go`の判定を残す。荷重主張表には主張、原典位置、三択の裁定を置く。`go`は未解決事項が公開条件に影響しない場合に限る。`no-go`は直前の調査、質問設計または公開説明の工程へ戻し、語尾や体裁だけを直す工程へ送らない。

## プロンプト断片

```text
次の成果物から、数値、日付、固有名詞、制度上の主張、引用を機械的に抽出してください。出力列は「主張｜種別｜出典注記あり？｜as_of｜evidence_id または NONE」です。evidence_id欠落、指定された鮮度条件超過、定義不一致は警告列に記載してください。重要度や採否は判断しないでください。
```

```text
次の公開予定稿を漏えい走査してください。個人情報、内部ID、内部リンク、ファイル名、メタデータ、文脈による再識別可能性を探し、「種類｜位置｜該当断片｜要確認理由」で返してください。安全だと推測して削除候補を確定しないでください。
```

## 成果物と完了条件

成果物は、対象版に対応する公開前レビュー記録である。全荷重主張に三択の裁定があり、未解決事項が明記され、議員本人の`go / no-go`判断と日付が記録された時点で完了する。`no-go`も、戻し先が決まっていればこの検算の完了である。

## 次のワークフローへの接続

`go`なら指定した提出または公開へ進む。質問への答弁を得た後は[07 答弁後追跡](07-track-after-answer.md)へ渡す。`no-go`なら荷重主張が生じた直前のワークフローへ戻し、根拠の追加、問いの再設計または表現の再構成を行う。

## アンチパターン

- **全文を同じ密度で校正する。** 表現の手直しに注意を奪われ、結論を支える荷重主張の検算が薄くなる。
- **AIに荷重主張を確定させる。** 質問や公開説明の責任を機械の重要度判定へ移してしまう。
- **出典名があるだけで通す。** 原典位置、時点、定義がなければ、その主張を再検証できない。
- **弱めることを敗北とみなす。** 根拠を超えた断定を残し、文章の強さを検証可能性より優先する。
- **不可を表面修正へ送る。** 前提の欠陥が残ったまま、読みやすい誤りへ整えてしまう。
