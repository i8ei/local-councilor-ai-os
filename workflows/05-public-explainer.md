# 05 公開説明

## 要旨

調査結果や議会でのやり取りを、特定の住民が理解し次の動きを追える公開説明へ変換する。媒体ごとに必要最小限の事実を選び、結論、根拠、今後の順で示す。AIは平易化、誤読候補の抽出、媒体への整形を担えるが、対象読者、政治的な調子、公開の最終判断は議員本人が決める。

## 発動条件

議会報告、SNS、広報原稿、説明会資料などの公開成果物を作るときに発動する。[証拠台帳](../way-of-working/06-evidence-ledger.md)に公開説明の主要事実が登録され、公開情報だけで説明の骨格を再構成できることを開始条件とする。

## 入力と前提

- 説明対象の案件サマリーと現在判断
- `verified`または必要に応じて`reconciled`となった証拠台帳
- 公開する結論と、その後に予定する確認または行動
- 媒体の文字数、掲載形式、公開範囲、公開予定日
- 作業先となる[公開前レビューテンプレート](../templates/public-output-review.md)

住民の声や現場観察を起点とした場合も、内部原典を公開主張の出典にしない。公開操作はこのワークフローに含めず、人による承認後に別途行う。

## 手順

1. **【人】読者と媒体を一つずつ固定する。** 誰に何を理解してほしいか、読後に何を確認できる状態にするかを一文で書く。議会報告、短文投稿、広報原稿、口頭説明では必要な前提と出典の見せ方が異なる。一つの文章を全読者へ流用せず、読者ごとに成果物を分ける。

2. **【併走】説明を支える最小事実を選ぶ。** 証拠台帳から、結論が依存する事実だけを抜き出し、各文に`evidence_id`を対応させる。数値は`value`、`as_of`、`definition`、`source`の4点を一組で保持する。本文に収まらない要素は脚注、出典欄、リンク先へ移してよいが、読者が同じ原典へ到達できる形を崩さない。

3. **【AI】平易な初稿を作る。** 最初に結論を一文で示し、次に根拠、最後に今後起きることと次回報告の時期を書く。制度用語は必要な範囲で言い換え、意味が変わる場合は原語を併記する。説明会資料や読み上げ原稿では、一文を短くし、声に出して息継ぎできるリズムへ整える。

4. **【併走】断定の強さを証拠と照合する。** 主語の範囲、因果関係、代表性、時点が、根拠の示す範囲を超えていないか一文ずつ確認する。分析は分析、議員の判断は判断と分かる表現にする。仮説は`仮説`または`要確認`と明記して残すか、公開文から外す。文章上の勢いを理由に仮説を断定へ変えない。

5. **【AI】誤読と反論の候補を洗い出す。** 読者が対象範囲を広く受け取る箇所、数値を別定義と混同する箇所、行政の行為と議員の判断を取り違える箇所を列挙する。文脈の組合せから個人や関係者を推測できる可能性も確認し、削除、一般化、注記の修正案を示す。

6. **【人】政治的な調子を確定する。** 批判は制度、判断過程、実施状況へ向け、人物の動機や人格を推測しない。行政側の説明、制約、未確定事項を公平に記述し、行政自身が読んでも事実関係を争う必要がない文章にする。その上で、議員としての評価と次に求める行動を明確にする。

7. **【人】公開前ゲートへ引き渡す。** [06 公開前検算](06-verify-before-output.md)の公開前ゲートを、[公開前レビューテンプレート](../templates/public-output-review.md)で実行する。誤読候補と修正結果もレビュー記録へ残す。結果が承認になるまで、投稿、配布、掲載、外部送信を行わない。

## プロンプト断片

```text
次の事実骨格と出典情報だけを使い、指定した読者と媒体向けの公開説明へ書き直してください。構成は「結論」「根拠」「今後起きること」「次に報告する時期」の順にしてください。入力にない事実、原因、評価は追加しないでください。追加したくなった箇所があれば、本文には入れず「不足情報」として理由を列挙してください。数値は値、時点、定義、出典の対応を維持してください。
```

```text
次の公開原稿について、誤読、過度な一般化、反論される弱点、文脈から個人を推測できる箇所を列挙してください。各項目に、問題となる文、根拠とのずれ、最小の修正案を示してください。新しい事実を補って修正しないでください。
```

## 成果物と完了条件

成果物は読者と媒体を明記した公開説明案と、その公開前レビュー記録である。次をすべて満たしたときに完了とする。

- 一つの読者と一つの媒体に合わせた原稿がある
- 主要事実が証拠台帳へ対応し、すべての数値が4点セットを保っている
- 結論、根拠、今後、次回報告時期が読み分けられる
- 誤読、反論、再識別の候補と修正結果が記録されている
- 公開前ゲートへの引継ぎが記録され、議員本人の承認待ちまたは承認済みの状態が明確である

## 次のワークフローへの接続

必ず[06 公開前検算](06-verify-before-output.md)へ渡し、公開前ゲートの承認後にのみ公開する。説明の中で新たな不一致や不足が見つかった場合は[03 公開調査](03-public-research.md)へ戻す。答弁や実施状況の次回報告を約束した場合は、[07 答弁後追跡](07-track-after-answer.md)の次回確認日と同期させる。

## アンチパターン

- **一つの原稿を全媒体へ流用する。** 読者に必要な前提と出典の粒度が合わず、短文化の過程で条件が落ちる。
- **分かりやすさのために数値の時点や定義を省く。** 別の数値との誤比較を招き、読者が検証できなくなる。
- **仮説を強い言葉へ言い換える。** 根拠が支えない因果や動機が、議員の断定として残る。
- **行政批判を人物批判にする。** 制度上の論点が感情的な対立へ置き換わる。
- **レビュー記録なしに公開する。** 根拠、再識別、公開範囲を誰が確認したか追跡できない。
