# 来歴契約

## 目的と参照実装

来歴は、公開原典の取得から正規化値までを辿り、派生 DB を作り直す記録である。`bootstrap/cli/db.py` の指標行と `modules/minutes_db/adapters/base.py` の取得結果を参照実装とする。前者は値、原表記、単位、時点、定義、原典位置、取得日時、検証状態を行単位で保持する。後者は要求 URL、解決後 URL、媒体型、文字コード、内容ハッシュ、キャッシュ、要求記録を保持する。

SQLite は原典ではない。修正は原典と変換処理へ反映して再生成する。

## 正規化行の必須フィールド

| フィールド | 型 | 説明 |
|---|---|---|
| `value` | number/string/null | 機械比較に使う正規化値 |
| `raw_value` | string | 原典にある変換前の表記 |
| `unit` | string | 原典の単位。値と分離する |
| `as_of` | string/object | 基準日または対象期間 |
| `definition` | string/object | 対象範囲、会計範囲、算定方法 |
| `source_name` | string | 公表資料またはデータセットの正式名 |
| `source_url` | string | 第三者が公開原典へ到達できるURL |
| `source_locator` | object | ページ、表、行、セル、発言番号などの原典内位置 |
| `fetched_at` | string | 原典取得時の ISO 8601 日時 |
| `verification_state` | enum | 検証の到達状態 |

複数の金額列がある表では、名前付きの各金額列が `value` の責務を持ち、原行全体を `raw_value` に保存してよい。欠損、ゼロ、非該当を同じ表現へ潰さない。単位換算、丸め、符号変換は `transformations` に入力、式、出力を残す。

`source_locator` は JSON として保存できる構造にし、少なくとも原典の一箇所へ戻れる情報を持つ。PDF は印刷ページと PDF ページ、表データはシートとセルまたは行、議事録は会議IDと発言IDを記録する。検索結果の順位だけを位置情報にしない。

## 取得層の来歴

取得単位では次を記録する。

| フィールド | 説明 |
|---|---|
| `requested_url` | 取得を開始した公開URL |
| `final_url` | リダイレクト後に実際に取得したURL |
| `request_time` | 各要求を送った日時 |
| `robots_decision` | 対象URLに対する `allowed`、`denied`、`not_applicable` の判断 |
| `cache_key` | 要求URLから決定的に作るキー |
| `from_cache` | ネットワーク取得かキャッシュ再利用か |
| `content_type` | 取得時の媒体型 |
| `encoding` | テキスト復号に使った文字コード |
| `sha256` | 取得内容のハッシュ |
| `requests` | リダイレクトを含む要求URL、応答URL、状態、日時の列 |

参照 fetcher のキャッシュキーは要求 URL の SHA-256 であり、本文と JSON メタデータを対で検証する。robots.txt はネットワーク要求の前に確認する。一般契約では、その確認結果も `robots_decision` として永続化し、後から「確認したはず」で済ませない。

公開する `source_url` と取得記録の URL に、ユーザー名、パスワード、API キー、AppId などの認証情報を含めない。認証が必要な取得設定は秘密管理へ分離し、公開来歴から参照しない。

## 派生と再構築

派生行は、元レコードIDの配列、変換名、ツール版、設定、実行日時、出力ハッシュを持つ。集計行は入力IDと決定的な式から同じ値を再計算できなければならない。原典から抽出した行を更新する場合は、取得内容のハッシュまたは版を比較し、黙って上書きしない。

DB を削除した状態から、保存済み原典、取得メタデータ、変換処理だけで同じスキーマと行を再生成する。手補正が必要なら、補正対象の原典位置、補正前の値、補正後の値、理由を入力データとして版管理する。

## 検証状態

正規語彙は[根拠データ契約](evidence_schema.md)と同じ `draft`、`discovered`、`verified`、`reconciled`、`rejected` とする。状態だけを変更せず、確認日時、確認者、実施した検査、未解決事項を伴わせる。

既存の bootstrap 実装にある `verified_source_extraction` は契約境界で `verified`、`needs_review` は問題を付けた `draft` へ正規化する。`reconciled` は同じ意味で使う。予算と決算など合計突合が必要なデータは、差額ゼロを確認するまで `reconciled` にしない。

## アンチパターン

- **公開URLへ認証情報を入れる。** 来歴の共有で秘密が漏れ、再利用者ごとの認証分離もできない。
- **DB行だけを手で直す。** 再構築すると修正が消え、なぜ値が変わったか追跡できない。
- **ページ番号だけを省く。** 原典全体を再読しないと転記を確認できず、位置情報の責務を果たさない。
- **検証状態だけを進める。** 実施した検査と確認結果がなく、状態名が自己申告になる。
