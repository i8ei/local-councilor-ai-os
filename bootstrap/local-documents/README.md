# Tier 3: 予算書と決算書などの地域固有文書

Tier 3 は、自治体ごとに版面が異なる予算書と決算書を、人が原本を見ながら検証可能な SQLite へ変換する手順である。全自治体へ通用する完全自動 OCR は対象外とする。この層は、人間併走を前提に、抽出の速さより原典位置と検算可能性を優先する。

格納形式と合計検算は[決算レビュー](../../modules/settlement_review/)を使う。SQLite は原典から再構築できる派生物であり、PDF の代替正本ではない。

## 取得前に候補だけ診断する

まず公式の予算・決算索引ページだけを取得し、リンク候補を確認する。

```sh
python3 -m bootstrap.cli.local_documents diagnose \
  --index-url 'https://www.example.jp/official/budget-and-settlement/'
```

このコマンドは索引HTMLだけを読み、候補文書を列挙して`needs_confirmation`で止まる。PDF、XLSX、CSV、ZIPは取得せず、SQLiteも作らない。利用者は`later`（後日）、`sample`（1〜3件テスト）、`targeted`（年度・会計・種類を限定）、`full`（必要範囲を全件）のいずれかを選ぶ。

## 候補を少数取得して品質を確認する

診断結果の1始まりの候補番号を指定し、最大3件だけ取得する。

```sh
python3 -m bootstrap.cli.local_documents sample \
  --index-url 'https://www.example.jp/official/budget-and-settlement/' \
  --candidate 1 \
  --candidate 3 \
  --output-dir /tmp/municipality-document-sample
```

`sample`は診断と同じ公式索引から観測したURLだけを取得し、原本path、SHA-256、
媒体型、PDF署名、取得日時、cache状態をJSONで返す。`pdftotext`がPATH上にあれば、
先頭3ページだけでテキスト層を確認し、文字数を記録する。数値抽出、SQLite作成、
検証状態の昇格は行わない。`--candidate`を省略した場合は`--limit 1`（最大3）で
先頭候補を選ぶ。

## 0. PDF の型を先に判定する

取得前後に、各 PDF について次を記録する。

1. テキスト層があるか。数ページを `pdftotext -layout` で出し、見出し、行、金額が読めるか確認する。
2. 画像だけの PDF か。文字が見えても選択と抽出ができなければ、画像経路へ送る。
3. 見開き構成か。印刷上の一ページが、PDF 上では左右二ページに分かれていないか確認する。
4. 原表の単位は何か。表題と各表頭を見て、行ごとに `円` や `千円` などの原単位を保持する。
5. 詳細表の開始 PDF ページはどこか。年度ごとに目視で特定する。

予算書は画像 PDF、決算書はテキスト層付き PDF であることが多い。同じ自治体や同じ年度だから同じ抽出方法だと決めない。

## 1. 公式サイトから原本を取得する

公式の索引ページから文書への経路を確認し、索引 URL と解決後の PDF URL を保存する。robots.txt を確認し、低頻度の直列取得とキャッシュを使う。再試行を短時間に繰り返さず、同じファイルを何度も取り直さない。取得日時、内容ハッシュ、媒体型、ファイル名を来歴へ残す。

公開 URL に認証情報や AppId を埋め込まない。公式ページから辿れない直リンクだけを根拠にせず、第三者が公表経路を確認できる状態にする。

## 2. 抽出経路を選ぶ

テキスト層が有効なら `pdftotext -layout` を基準にする。列位置を固定幅の窓として読み、年度ごとに開始ページと表頭を人が確認する。列窓が流用できても、見出しの意味まで前年と同じだとは推定しない。

画像 PDF は、ページ画像を見ながら視覚支援付きで転記する。ページ単位の並列バッチに分ける場合は、前後に一ページ以上の重なりを持たせ、結合時に印刷ページ番号と行キーで境界重複を除く。各バッチの先頭と末尾は人が原本と照合する。OCR の信頼度だけで採否を決めない。

見開き PDF は、左右を独立した表として確定しない。左ページの款、項、目、節と予算現額を、右ページの支出済額、翌年度繰越額、不用額へ、印刷ページ下部の番号順で組み直す。

## 3. 正規化して格納する

`modules/settlement_review/schema.sql` を適用し、総括表、歳入明細、歳出明細へ格納する。数値は原表記を `raw_value` に残し、正規化値とは分ける。`unit`、`as_of`、定義、出典名、URL、原典位置、取得日時、検証状態を各行へ持たせる。原典位置には `print_page` と `pdf_page` の両方を入れる。

歳入は款と項、歳出は款、項、目、節を識別する。解釈や質問候補は DB に書かず、検算後の分析成果から証拠台帳と判断ノートへ渡す。

## 4. 差額ゼロの検算ゲートを通す

`verify_totals.py` で、歳入の項合計と総括表の款、歳出の節合計と目、重複を除いた目合計と総括表の款を突合する。収入済額、不納欠損額、収入未済額、支出済額、翌年度繰越額、不用額、予算現額を対象にし、すべての差額がゼロであることを要求する。

単位が異なる行は比較せず、不一致として止める。単位換算は比較処理で明示し、入力行を暗黙に書き換えない。差額が残る場合は原典位置へ戻り、抽出、ページ結合、粒度、単位を修正してから再実行する。

## 5. 検証済み DB と呼べる条件

SQLite の整合性確認だけでは足りない。歳入の款、歳出の款、歳出の節から目まで、必要な全階層で差額ゼロになり、原典位置と単位を人が確認した後に限り、検証済み DB と呼ぶ。途中データは `draft` または `verified` に留め、合計突合を通った行だけを `reconciled` とする。

## 既知の罠

1. **文字が見えるだけでテキスト PDF と決める。** 画像上の文字にはテキスト層がなく、テキスト抽出経路では空欄や列崩れになる。
2. **見開きの左右を別レコードにする。** 款項目節と予算現額が左、執行額と不用額が右に分かれる場合は、印刷ページ番号順に対で結合する。
3. **単位を暗黙にそろえる。** 予算書が千円、決算書が円の場合がある。単位は行に保存し、比較時だけ明示換算する。
4. **節行の目合計をそのまま足す。** 各節行に同じ目合計が反復される表では、款、項、目で一度重複排除してから集計する。
5. **繰越明許の小区画を別の目にする。** 同じ目番号の繰越分は親の目へ統合し、項合計と一致するまで目の区画を累積する。
6. **改ページごとに新しい目を開く。** 継続ページでは款項目の文字だけが再掲され、金額がないことがある。款または項が実際に変わった場合だけ新規レコードを開く。
7. **前年の開始ページと列位置を無条件に使う。** 年度で PDF 生成器や版面は変わる。列窓を再利用できても、開始ページは PDF ごとに目視で決める。
8. **当初予算と決算だけを比べる。** 決算書の予算現額を使えば、当初予算、補正後の予算現額、決算の三段階を比較できる。結合前に単位と会計範囲を確認する。
