# 複数AIとの協働

## 要旨

複数AIは、同じ仕事を人数分だけ速くする仕組みではない。文脈を保持して統合する役、実装する役、独立に疑う役、公開情報を探す役を分け、互いの誤りを止める仕組みである。委任によって増やすのは作業量であり、判断主体ではない。すべての出力は作業途中の成果物として扱い、採用、コミット、提出、公開の最終判断は議員本人に残す。

## 役割設計

オーケストレーターAIを一つ置く。オーケストレーターは案件の文脈、正本ノート、根拠の検証状態、公開と内部の境界を保持し、判断材料の整理、成果物の統合、議員本人との対話を担う。委任先へ渡す情報を選び、戻ってきた結果の矛盾と欠落を確認するが、政策上の優先順位や賛否は決めない。

実装エンジンは、指定された範囲の文書、処理、テストを作る。分析エンジンは、比較、集計、構造化、反証候補の抽出を担う。研究エンジンはウェブ上の公開原典を探索し、主張ごとに出典URLを返す。役割は機能で決め、特定の提供元や製品名に結び付けない。同じAIでも呼び出しと文脈を分ければ別の役を担えるが、独立性が必要な検証では実装時の会話を引き継がせない。

これらの上に議員本人がいる。AI同士の一致は採用理由にならない。何を問い、何を事実として採り、どう説明するかは、[個人の作業美学](../way-of-working/10-personal-operating-style.md)に従って議員本人が引き受ける。

## 委任の原則

委任は、分割可能な並列バッチ、長時間の実装、独立した調査で、依頼文の作成と統合レビューを含めても明確に得をするときに使う。単一ファイルの編集、短い確認、その場で返せる判断補助はオーケストレーターが直接扱う。小さな仕事まで委任すると、説明と再統合の費用が成果を上回る。

委任文には、少なくとも次の契約を入れる。

- **範囲。** 触れてよいファイル、読んでよい資料、対象外、期待する成果物を明記する。
- **形式。** 言語、語調、見出し、文字数、表記、例の示し方、禁止表現を固定する。
- **正直さ。** 未検証は未検証と記し、見つからないものは不明と答え、数値、引用、URL、実行結果を作らない。
- **状態変更。** コミット、公開、外部送信を行わない。必要な操作が範囲外なら実行せず、理由と必要な権限を返す。

委任先の完了報告は検証の入口であり、完了の証明ではない。オーケストレーターは差分、テスト結果、根拠、範囲逸脱を確認し、コミットまたは公開の前に議員本人へ判断可能な形で示す。

## 検証の分離

レビュー担当を実装担当と同一にしない。実装者は、自分が想定した成功経路を再確認しやすく、同じ前提の誤りを見落とすからである。AI同士でも、[検算ゲート](../way-of-working/07-verification-gates.md)を次の形で通す。

1. 実装エンジンが成功を報告しても、オーケストレーターが同じテストを自分で実行し、終了状態と出力を確認する。報告文や貼り付けられたログだけでは通さない。
2. ライブ環境に関する主張は、実装者が触れていない別の事例で再検証する。例：実装者が一つの自治体で確認した取込処理は、オーケストレーターが別の自治体で実行し、同名候補が複数ある経路、欠測、未対応形式などの端の経路も確認する。
3. 研究結果は、出典URLの一部をオーケストレーターが直接取得し、主張、対象時点、原典内の位置が一致するかを確認する。URLが実在するだけでは採用しない。

検証で不一致が出た場合は、平均を取らず、どの前提、入力、定義が違うかを記録する。解消できなければ不明のまま止める。

## 合議パターン

重要な設計判断は「提案、批判、決定」の三段階に分ける。エンジンAへ制約と目的を渡して案を作らせる。エンジンBにはAの結論を正解として渡さず、独立に考えたうえで穴、反例、運用負担、壊れる条件を探すよう指示する。最後にオーケストレーターが相違点と選択肢を整理し、議員本人が決める。

同じ親会話から派生し、同じ推論や要約を共有した補助AIは、数が増えても独立した反対意見にはならない。共有文脈は作業分担に向く。前提を疑うレビューは、別の呼び出しで必要最小限の事実と評価基準だけを渡して行う。

## 並列バッチ委任

大量の執筆や実装は、成果物を独立したバッチへ分ける。全バッチに共通の形式契約、用語、検証方法を渡し、各バッチには触れてよいファイルを排他的に割り当てる。共通ファイルは一つのバッチだけが所有し、他のバッチは変更案だけを返す。

並列化すると、見出しの粒度、言い切りの強さ、例の置き方にずれが出る。これは偶発的な失敗ではなく、繰り返し発生する統合費用である。オーケストレーターは最初から統合レビューの時間を確保し、用語、重複、リンク、文体を全体でそろえる。各バッチの局所的な完成を、そのまま全体の完成としない。

## 研究エンジンの規律

研究結果は主張ごとに出典URL、確認日、原典内の位置を伴わせる。公式原典へ到達できない場合は不明と書く。検索結果、要約、二次資料は原典候補を見つける手掛かりであり、独立確認が終わるまでは事実ではない。

例：研究エンジンがベンダーのテナントURLを返しても、その文字列だけで対応先と確定しない。オーケストレーターが許可された経路から直接取得し、対象名、ページ内容、転送先を確認してから採用する。取得が禁止または不能なら、ライブ未検証として残す。

## 失敗の還流

タイムアウト、範囲逸脱、バッチ間の文体差、架空の数値やURLは、個別の注意で閉じない。発動条件、症状、見逃した理由、次回の停止条件を記録し、委任文の定型、受入検査、範囲ロックへ戻す。この流れは[失敗駆動の運用](../way-of-working/09-failure-driven-operations.md)を委任へ適用したものである。失敗を隠さず、次の一回を短い契約と検査で止める。

## アンチパターン

- **実装者の成功報告を検証完了とする。** 同じ前提と成功経路をなぞるだけになり、端の経路が未確認のまま残る。
- **委任のために小さな仕事まで分解する。** 説明、待機、統合の費用が直接作業を上回る。
- **共有文脈から出た複数案を独立意見と数える。** 最初の要約に含まれた偏りを全員が継承する。
- **URLのない調査結果を事実として採る。** 原典へ戻れず、時点、定義、対象の取り違えを検算できない。
- **並列バッチを結合してそのまま完成とする。** 既知の文体差と用語差を、読者と後続作業へ押し付ける。
