# way-of-working 索引

`way-of-working/` は、地方議員がAIとデータを使って働くときの運用設計をまとめた11章である。個々のコマンドやモジュールの使い方ではなく、それらを安全に成立させる考え方を定める。実装の手順は[`workflows/`](../workflows/)、判断責任と安全境界の憲章は[`principles/`](../principles/)にある。

## 読み方

上から順に読むと、記録の作り方から対外的な説明、その後の追跡までが一続きの弧になる。

1. **土台をつくる**（01〜03） — 現在地を一本化し、データと判断を分ける。
2. **出典を裁定する**（04） — 用途ごとに正しい入口を固定する。
3. **公開できる形にする**（05〜06） — 内部情報を問いへ組み替え、原典まで戻れる形で台帳化する。
4. **止める**（07） — 通告前・公開前・保存前で荷重主張を検算する。
5. **働きかける**（08、11） — 答弁で終わらせず実装まで追跡し、困りごとを制度の効く一点へ変換する。
6. **運用を保つ**（09〜10） — 失敗を仕組みへ変え、考える力と休む余地を守る。

各章は要旨・本論・最小の始め方・アンチパターンで構成する。章どうしは本文中で相互に参照する。

## 章一覧

- **[01 正本主義](01-canonical-notes.md)** — 一論点の現在地は正本ノート一本に集約し、過去・スナップショット・チャット履歴を判断の保管場所にしない。
- **[02 MOCとライフサイクル](02-moc-and-lifecycle.md)** — 案件ごとのMOCを入口に、ノートの存在と検索露出を分け、記録を残したまま日常検索の雑音を減らす。
- **[03 データと判断（二層構造）](03-data-vs-judgment.md)** — 一次データはSQLite/FTS5、解釈と判断はObsidian。この境界は便宜ではなく正本性の境界である。
- **[04 Authority Router](04-authority-router.md)** — 同じ指標でも用途で正しい出典が変わる。回答前に必ず「指標×用途」の経路を引き、値は複製しない。
- **[05 内部情報から公開可能な問いへ](05-internal-to-public.md)** — 住民の声を匿名化して移すのではなく、公開原典だけで検証できる問いへ組み替える。
- **[06 証拠台帳](06-evidence-ledger.md)** — 外部で使う主張は原典の該当箇所まで戻れる形で台帳化する。台帳だけで骨格を再生できない成果物は外へ出せない。
- **[07 検算ゲート](07-verification-gates.md)** — 荷重主張を抜き出し、原典・定義・時点・計算まで検算する。通告前・公開前・保存前で止める事故が違うため、別々のゲートを置く。
- **[08 政策実装トラッカー](08-policy-implementation-tracker.md)** — 答弁・期限・予算化・実施・測定・次の質問の連鎖を、一案件一ノートで複数年度にわたり追う。
- **[09 失敗駆動の運用](09-failure-driven-operations.md)** — 起きた事故を即時修正→一般化したルール→自動ガード→標準運用への編入の順で処理する。
- **[10 個人の作業美学](10-personal-operating-style.md)** — 小さく作るが正本・出典・承継リンクは雑にしない。仕組みは生産量ではなく、考える力と休める余地を守るために置く。
- **[11 ツボ探し](11-advocacy-routing.md)** — 困りごとを批判や単発要望で終わらせず、制度の最も効く一点へ働きかけ、副作用込みの最小の手当てにする。

## コードとの対応

一部の章は`lcaios/`の読み取り専用CLIとして機械化されている。04の回答前ゲートと鮮度条件は`lcaios status`／`freshness`、05〜07の公開前検査は`lcaios verify output`、03のデータ層の整合は`lcaios verify database`が補助する。CLIは検査と道案内だけを行い、判断と公開は議員本人に残す。次の一手は[`lcaios doctor`](../lcaios/)がまとめて示す。
